## Overview

Chirpy is a fully functional backend API for a Twitter-like microblogging platform. It handles user registration, JWT-based authentication, short-form posts ("chirps") with server-side content moderation, a token refresh and revocation system, and a webhook-driven premium subscription tier — all backed by PostgreSQL with schema managed through Goose migrations and type-safe queries generated by `sqlc`. The entire server is built on Go's standard `net/http` package with no routing framework.

---

## Key Features

- 🔐 **Dual-Token Auth** — Short-lived JWT access tokens (1-hour expiry) paired with long-lived refresh tokens (60-day expiry), fully persisted and revocable in the database
- 🔑 **Argon2id Password Hashing** — Memory-hard, industry-standard password hashing via `alexedwards/argon2id` with default hardening parameters
- 🐦 **Chirp CRUD** — Create, list (all or by author), fetch by ID, and delete chirps capped at 140 characters; list endpoint supports `?author_id=<uuid>&sort=desc` query parameters
- 🧹 **Content Censorship** — Server-side word filter replaces configured profane terms with `****` before any chirp is persisted
- 🔴 **Chirpy Red (Premium Tier)** — Webhook endpoint `/api/polka/webhooks` upgrades a user's `is_chirpy_red` flag, secured by an `ApiKey` header validated against a configured key
- 📊 **Admin Metrics** — Atomic hit counter middleware tracks fileserver requests; exposed as an HTML admin dashboard at `/admin/metrics`
- 🏥 **Health Check** — `GET /api/healthz` for uptime monitoring
- 🗄️ **Type-Safe SQL** — All database interactions are code-generated by `sqlc` from raw `.sql` query files; no ORM

## Installation

**Prerequisites:** Go 1.22+, PostgreSQL, [`goose`](https://github.com/pressly/goose), [`sqlc`](https://sqlc.dev)

```bash
# 1. Clone the repository
git clone https://github.com/your-username/chirpy.git
cd chirpy

# 2. Install Go dependencies
go mod download

# 3. Create a .env file with your configuration
cat > .env << 'EOF'
DB_URL=postgres://username:password@localhost:5432/chirpy?sslmode=disable
PLATFORM=dev
JWT_SECRET=your-super-secret-signing-key
POLKA_KEY=your-webhook-api-key
EOF

# 4. Run database migrations
goose -dir sql/schema postgres "postgres://username:password@localhost:5432/chirpy?sslmode=disable" up

# 5. Build the binary
go build -o chirpy .
```

# Start the server (listens on :8080)

### API Endpoints

**Authentication & Users**
```
POST /api/users            Register a new user (email + password)
POST /api/login            Authenticate; returns access token + refresh token
POST /api/refresh          Exchange a valid refresh token for a new access token
POST /api/revoke           Revoke a refresh token (Bearer token required)
PUT  /api/users            Update email and password (Bearer token required)
```

**Chirps**
```
POST   /api/chirps                  Create a chirp, max 140 chars (Bearer token required)
GET    /api/chirps                  List all chirps; optional: ?author_id=<uuid>&sort=asc|desc
GET    /api/chirps/{chirpID}        Fetch a single chirp by UUID
DELETE /api/chirps/{chirpID}        Delete a chirp (author only, Bearer token required)
```

**Webhooks & Admin**
```
POST /api/polka/webhooks   Upgrade user to Chirpy Red (ApiKey header required)
GET  /admin/metrics        View fileserver hit counter (HTML response)
POST /admin/reset          Reset hit counter + delete all users (PLATFORM=dev only)
GET  /api/healthz          Health check; returns 200 OK
```

**Example — Register, login, and post a chirp:**

```bash
# 1. Register a user
curl -s -X POST http://localhost:8080/api/users \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com","password":"hunter2"}'

# 2. Login to receive tokens
TOKEN=$(curl -s -X POST http://localhost:8080/api/login \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com","password":"hunter2"}' \
  | jq -r '.token')

# 3. Post a chirp
curl -s -X POST http://localhost:8080/api/chirps \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"body":"Hello, Chirpy!"}'
```

## Project Structure

```
chirpy/
├── main.go                              # Entry point; loads .env, opens DB, starts mux
├── server_mux.go                        # HTTP mux setup; registers all routes on :8080
├── middleware_and_stateful_handlers.go  # All HTTP handlers + metrics middleware
├── censor.go                            # Profanity filter for chirp bodies
├── resp_w_json.go                       # JSON response helper (sets Content-Type + status)
├── resp_w_error.go                      # JSON error response helper
├── index.html                           # Static file served under /app/
├── assets/
│   └── logo.png
├── internal/
│   ├── auth/
│   │   ├── API.go        # ApiKey extraction from Authorization header
│   │   ├── JWT.go        # MakeJWT / ValidateJWT (HS256) + GetBearerToken
│   │   ├── passwords.go  # HashPassword / CheckPasswordHash (argon2id)
│   │   └── tokens.go     # MakeRefreshToken (32-byte cryptographically random hex)
│   └── database/         # sqlc-generated database layer (DO NOT EDIT)
│       ├── db.go         # DBTX interface + Queries constructor
│       ├── models.go     # User, Chirp, RefreshToken structs
│       ├── users.sql.go
│       ├── chirps.sql.go
│       └── tokens.sql.go
└── sql/
    ├── queries/          # Raw SQL consumed by sqlc
    │   ├── users.sql
    │   ├── chirps.sql
    │   └── tokens.sql
    └── schema/           # Goose migration files
        ├── 001_users.sql
        ├── 002_chirps.sql
        ├── 003_passwords.sql
        ├── 004_tokens.sql
        └── 005_red.sql       # Adds is_chirpy_red boolean to users
```

## Contributing

Contributions are welcome. Please fork the repository, create a feature branch, and open a pull request. New routes should be registered in `server_mux.go` with corresponding handler methods on `*apiConfig`. Any schema changes must be accompanied by a new numbered Goose migration file in `sql/schema/`, followed by a corresponding query in `sql/queries/` and a `sqlc generate` run.
